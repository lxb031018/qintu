## 🔗 功能一：绑定管理

### 业务目标
建立发送者与接收者之间的绑定关系，使发送者能够为接收者规划并发送导航任务。

### 绑定规则

| 规则 | 说明 |
|------|------|
| **绑定方式** | 手机号绑定（发送者输入接收者手机号） |
| **身份说明** | 发送者必须先填写自己的姓名，接收者将看到此信息 |
| **确认机制** | 单方发起 + 接收者确认（方案 B） |
| **绑定上限** | 发送者最多绑定 5 个接收者，接收者最多绑定 3 个发送者 |
| **绑定状态** | `pending`（待确认）→ `active`（已激活）→ `revoked`（已解除） |

### 🔒 隐私保护规则

| 规则 | 说明 |
|------|------|
| **不获取发送者位置** | 亲途完全不需要获取发送者（子女）的位置信息，保护发送者隐私 |
| **输入框默认隐藏** | 所有手机号输入框默认用圆点（●）隐藏数字 |
| **眼睛图标切换** | 点击"眼睛"图标显示真实号码 2 秒，然后自动恢复圆点 |
| **界面脱敏显示** | 所有界面显示的手机号必须脱敏：`138****5678`（中间 4 位隐藏） |
| **输入时完整显示** | 仅在用户点击"眼睛"后短暂显示 2 秒（用于校验） |
| **日志不打印** | 所有日志中不得出现完整手机号，必须脱敏后打印 |
| **API 传输加密** | 手机号通过 HTTPS 传输，后端存储时加密 |
| **接收者可见** | 接收者只能看到发送者的脱敏手机号，无法获取完整号码 |
| **发送者可见** | 发送者只能看到自己输入的完整号码，列表中也显示脱敏号码 |

---

### 📍 位置获取异常处理

| 异常场景 | 处理方式 | 发送者端提示 |
|---------|---------|-------------|
| **接收者位置获取失败** | 无法规划路线，起点未知 | "无法获取接收者当前位置，请确保接收者已开启定位并尝试重新发送" |
| **接收者定位未开启** | 发送者端提示接收者开启定位 | "接收者未开启定位，请通知接收者在手机设置中开启位置服务" |
| **接收者网络离线** | 位置数据过期（超过 5 分钟） | "接收者网络可能不稳定，建议稍后重试或电话联系确认" |
| **接收者拒绝位置共享** | 权限被拒绝 | "接收者未授权位置权限，请在绑定关系中确认定位已开启" |

**降级策略**：
1. **首次失败**：自动重试 1 次（间隔 3 秒）
2. **重试失败**：提示发送者上述异常信息
3. **建议操作**：提供"联系接收者"快捷拨号按钮
4. **不获取发送者位置**：任何情况下都不使用发送者位置作为替代起点

### 详细步骤说明

#### 步骤 1：发送者发起绑定请求

**发送者端 UI 流程**：

```
┌─────────────────────────────┐
│  ← 返回    绑定管理         │
├─────────────────────────────┤
│                             │
│  ┌─────────────────────┐   │
│  │  + 绑定新用户       │   │  ← 点击此处
│  └─────────────────────┘   │
│                             │
│  ── 已绑定的接收者 ──         │
│  138****5678  [已激活]      │
│  139****1234  [已激活]      │
│                             │
└─────────────────────────────┘

            ↓ 点击后弹出对话框

┌─────────────────────────────┐
│  发送绑定请求               │
├─────────────────────────────┤
│                             │
│  您的姓名（必填）           │
│  ┌─────────────────────┐   │
│  │ 张三                │   │  ← 填写自己的姓名
│  └─────────────────────┘   │
│                             │
│  对方手机号（必填）         │
│  ┌─────────────────────┐   │
│  │ 13800138000         │   │  ← 输入对方 11 位手机号
│  └─────────────────────┘   │
│                             │
│  对方将看到您的姓名和手机号  │
│  请确认信息真实有效          │
│                             │
│  [ 取消 ]    [ 发送请求 ]   │
│                             │
└─────────────────────────────┘
```

**发送者端逻辑**：
1. 用户点击"绑定新用户"
2. 弹出绑定请求对话框
3. **必填：填写自己的姓名**（用于接收者识别身份）
4. 输入对方 11 位手机号，校验格式
5. 调用 API：`POST /api/bindings/request-phone`
   ```json
   {
     "sender_name": "张三",
     "receiver_phone": "13800138000"
   }
   ```
6. 显示加载状态
7. 成功：提示"绑定请求已发送，等待对方确认"
8. 失败：提示错误原因（用户不存在、已达上限等）

**重要说明**：
- 姓名必填，否则无法发送请求
- 姓名用于接收者识别发送者身份
- 建议从用户资料中自动填充，允许修改

3. 点击通知图标，进入绑定请求列表
4. 显示请求列表（**重点显示发送者姓名**）
5. **接收者通过姓名判断是否认识发送者**

#### 步骤 3：接收者确认/拒绝绑定

**接收者端操作**：

```
            ↓ 点击"确认"后弹出确认对话框

┌─────────────────────────────┐
│  绑定请求确认               │
├─────────────────────────────┤
│                             │
│  您收到一个绑定请求         │
│                             │
│  发送者信息：               │
│  ┌─────────────────────┐   │
│  │ 姓名：张三           │   │  ← 重要！识别身份
│  │ 手机：138****5678   │   │  ← 脱敏显示
│  └─────────────────────┘   │
│                             │
│  绑定后，张三即可为您        │
│  规划路线并发送导航          │
│                             │
│  [ 拒绝 ]    [ 确认绑定 ]   │
│                             │
└─────────────────────────────┘
```

**接收者端逻辑**：
1. 用户点击"确认"
2. 弹出确认对话框，**重点显示发送者姓名和手机号**
3. **接收者通过姓名确认是否认识此人**
4. 调用 API：`POST /api/bindings/confirm-request`
   ```json
   {
     "request_id": "req_12345"
   }
   ```
5. 成功：
   - 提示"绑定成功，张三现在可以为您发送导航"
   - 刷新列表，该请求移至"已绑定"区域
6. 拒绝：
   - 调用 API：`POST /api/bindings/reject-request`
   - 提示"已拒绝张三的绑定请求"
   - 从列表中移除

**重要说明**：
- 接收者**必须能看到发送者姓名**才能决定是否接受
- 姓名是接收者判断是否信任发送者的关键依据
- 如果姓名不明确，接收者应拒绝请求

## 🗺️ 功能二：路线规划与发送

### 业务目标
发送者为接收者规划导航路线，并推送导航任务，接收者接受后执行导航。

### 导航任务状态

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| `pending` | 待接收 | 发送者发送导航任务 |
| `active` | 导航中 | 接收者接受导航 |
| `finished` | 已完成 | 到达目的地或接收者完成 |
| `cancelled` | 已取消 | 发送者或接收者取消 |

1. 获取接收者实时位置作为导航起点
   - 通过 API 查询接收者最新上报的位置
   - 若接收者位置不可用，则使用地图默认值
2. 调用高德路线规划 API
   ```json
   {
     "origin": "116.404,39.915",  // 接收者当前位置
     "destination": "116.428,39.929",  // 目的地
     "strategy": "0"  // 路线策略（最短时间/最少费用等）
   }
   ```
3. 渲染地图，显示路线
3. 显示路线摘要（距离、时间、主要道路）
4. 点击"发送导航给接收者"
   - 调用 API：`POST /api/tasks`
   ```json
   {
     "receiver_phone": "13800138000",
     "end_name": "北京站",
     "end_latitude": 39.929,
     "end_longitude": 116.428,
     "route_data": { /* 高德路线数据 JSON */ },
     "distance_meters": 15300,
     "duration_seconds": 1920
   }
   ```
5. 发送成功 → 进入监护页面

## 🔌 技术实现要点

### 1. 实时通信机制

| 功能 | 技术方案 | 说明 |
|------|---------|------|
| **绑定请求通知** | WebSocket / MQTT | 接收者实时收到绑定请求 |
| **导航任务推送** | WebSocket / MQTT + 系统推送 | 接收者实时收到导航任务 |
| **位置实时传输** | WebSocket | 仅在发送者查看位置时建立连接 |
| **路线更新** | WebSocket | 实时推送新路线 |
| **结束导航** | WebSocket | 实时推送结束指令 |

### 2. 高德地图集成

| 功能 | API/组件 | 说明 |
|------|---------|------|
| **POI 搜索** | `AMapPOISearch` | 目的地搜索 |
| **路线规划** | `AMapRouteSearch` | 驾车/步行路线规划 |
| **地图渲染** | `AMapView` | 显示地图和路线 |
| **导航组件** | `AMapNavi` | 实时语音导航 |
| **定位** | `AMapLocation` | 获取用户当前位置 |

### 3. 前端保活策略

| 平台 | 保活方式 | 配置 |
|------|---------|------|
| **Android** | Foreground Service | `flutter_background_service` |
| **iOS** | Background Modes | Location + Audio |

### 4. 数据流转

